# Repository Analysis: C1.2 — Repository Contracts

## 1. Assessment
The domain model (C1.1) defines `Account`, `Category`, and `Transaction` entities. Currently, only `TransactionRepository.gs` exists as a persistence interface. To complete the domain persistence layer, `AccountRepository` and `CategoryRepository` must be established.

## 2. Architecture Review
Repository contracts must remain storage-agnostic. They represent the domain's *need* for persistence, not the persistence implementation itself.
- **Goal**: Standardize CRUD operations (`save`, `findById`, `findAll`, `update`) across all repository types.
- **Domain-Specifics**: While basic CRUD is generic, some repositories may later require domain-specific lookups (e.g., `findByCategoryName`). These will be added when business requirements necessitate them, following the "no overengineering" constraint.
- **Consistency**: All repositories will follow the storage-agnostic interface established in C1.1.

## 3. Risks
- Leakage of storage concepts (e.g., Sheets/Rows) into the interfaces.
- Redundant interface definitions. Mitigation: Use a shared base interface definition if language/framework supports it, or strictly enforced naming conventions.

## 4. Acceptance Criteria
- [ ] `AccountRepository` and `CategoryRepository` interfaces are defined.
- [ ] `TransactionRepository` is consistent with the new repository interfaces.
- [ ] All repository interfaces support `save`, `findById`, `findAll`, `update`.
- [ ] No `delete` operations exist (enforcing immutability).
- [ ] No persistence implementation (SpreadsheetApp, etc.) introduced.

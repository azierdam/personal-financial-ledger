# Repository Analysis: C1.2 — Repository Contracts

## 1. Assessment
The domain model (C1.1) defines `Account`, `Category`, and `Transaction` entities. To complete the domain persistence layer, `AccountRepository` and `CategoryRepository` must be established, and `TransactionRepository` must be reviewed for lifecycle compliance.

## 2. Architecture Review
Repository contracts must remain storage-agnostic. They represent the domain's *need* for persistence, not the persistence implementation itself.
- **Repository Contract Standard**: All repositories follow a standardized contract:
  - **Naming**: `[EntityName]Repository`
  - **Semantics**: `save`, `findById`, `findAll` (required for all). `update` (optional, based on mutability).
  - **Responsibilities**: Repositories only persist/retrieve. Business lifecycle logic is excluded.
- **Immutability Principle**: 
  - `TransactionRepository`: Excludes `update` and `delete` (Historical events).
  - `AccountRepository`/`CategoryRepository`: Include `update` (Evolving configuration).

## 3. Implementation Approach
- Define `AccountRepository` and `CategoryRepository` with standard CRUD (minus lifecycle business logic).
- Remove `update` from `TransactionRepository` to enforce immutability.
- Define the Repository Contract Standard in a new governance document.

## 4. Acceptance Criteria
- [ ] `AccountRepository` and `CategoryRepository` interfaces are defined.
- [ ] `TransactionRepository` excludes `update` and `delete` methods.
- [ ] Repository Contract Standard is documented.
- [ ] Unit tests verify CRUD method availability/non-availability per contract.
- [ ] No persistence implementation (SpreadsheetApp, etc.) introduced.

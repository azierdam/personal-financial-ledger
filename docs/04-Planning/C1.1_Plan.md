# Implementation Plan: C1.1 — Transaction Domain Model

## Objective
Refine the core domain model to support robust financial processing, ensuring storage-agnostic design and defining repository interfaces that honor immutability.

## Key Changes
- `src/domain/Account.gs`: New entity definition.
- `src/domain/Transaction.gs`: Refactor to use Account/Category entities and Money value object.
- `src/repository/TransactionRepository.gs`: Define interface for storage operations (save, findById, findAll, update). Note: `delete` is excluded to enforce immutability.
- `test/unit/TransactionDomainTest.gs`: Add unit tests for entity structures and repository contract.

## Implementation Steps
1. Create `Account.gs` entity.
2. Refactor `Transaction.gs` to reference `Account` and `Category` with stable identifiers.
3. Update `TransactionRepository.gs` interface to remove business lifecycle methods (like `delete`).
4. Implement unit tests to ensure entities and interfaces are structured correctly.
5. Update `docs/01-Architecture/Domain_Model.md` to reflect the Ledger aggregate concept.

## Verification
- Unit tests verify entity field types and repository interface structure.
- Review domain model documentation against implemented entities.

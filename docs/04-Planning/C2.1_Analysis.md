# Repository Analysis: C2.1 — Google Sheets Repository

## 1. Assessment
The domain layer (C1.1/C1.2) defines `Account`, `Category`, and `Transaction` entities. This milestone implements these contracts using Google Sheets as the storage engine via an isolated infrastructure layer.

## 2. Architecture Review
- **Infrastructure Layer**: Isolated in `src/infrastructure/sheets/`.
- **SheetsGateway**: Lightweight component for SpreadsheetApp interaction.
- **Mapper Design**: Explicit mapping classes (`AccountMapper`, `CategoryMapper`, `TransactionMapper`) for domain-to-row transformation.
- **Repository Implementation**: Concrete classes implementing repository contracts.
- **Error Handling**:
  - `EntityNotFoundException`: Thrown when a record does not exist.
  - `DuplicateEntityException`: Thrown when an ID collision occurs.
  - `MissingWorksheetException`: Thrown when the required sheet is unavailable.
  - `MalformedDataException`: Thrown when sheet row data does not conform to expectations.
  - `SpreadsheetIOException`: Encapsulates I/O failures.

## 3. Risks
- **Performance**: High I/O frequency. Mitigation: Batching is out of scope (future optimization).
- **Coupling**: Storage structures leaking. Mitigation: Strict usage of Mappers and `SheetsGateway`.

## 4. Acceptance Criteria
- [ ] `SheetsGateway` implemented for spreadsheet access.
- [ ] Explicit mappers created for all entities.
- [ ] Concrete repositories implemented with storage-agnostic contracts.
- [ ] All error cases documented in the plan are handled.
- [ ] Integration tests verify repository CRUD without leaking storage concerns.
- [ ] No domain layer changes.

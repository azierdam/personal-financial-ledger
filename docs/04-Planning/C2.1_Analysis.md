# Repository Analysis: C2.1 — Google Sheets Repository

## 1. Assessment
The domain layer (C1.1/C1.2) defines `Account`, `Category`, and `Transaction` entities with associated repository contracts. This milestone requires implementing these contracts using Google Sheets as the storage engine.

## 2. Architecture Review
The persistence layer must be strictly isolated in the infrastructure layer to prevent `SpreadsheetApp` dependency leakage into the domain.
- **Infrastructure Layer**: Create `src/infrastructure/sheets/` for persistence implementations.
- **Mapping Strategy**: Implement specialized `Mapper` classes to transform domain objects to/from Sheet row arrays.
- **Repository Implementation**: Create concrete `SheetsAccountRepository`, `SheetsCategoryRepository`, and `SheetsTransactionRepository` that implement the interfaces defined in `src/repository/`.
- **Error Handling**: Standardize exceptions for missing entities or I/O failures, keeping storage-specific errors encapsulated.

## 3. Risks
- **Performance**: High frequency of I/O operations with `SpreadsheetApp`. Mitigation: Batch read/write operations (deferred to future optimization sprint).
- **Coupling**: Storage structures leaking into domain logic. Mitigation: Strict enforcement of Mapper pattern.

## 4. Acceptance Criteria
- [ ] Concrete implementations of `AccountRepository`, `CategoryRepository`, `TransactionRepository` using Google Sheets exist.
- [ ] Persistence logic is encapsulated in `src/infrastructure/sheets/`.
- [ ] Domain layer remains unchanged.
- [ ] Integration tests verify storage/retrieval against Sheets.
- [ ] Immutability is preserved (no `delete` implementation for `Transaction`).
- [ ] `SpreadsheetApp` is NOT referenced in `src/domain/` or `src/repository/` (except the implementation classes).

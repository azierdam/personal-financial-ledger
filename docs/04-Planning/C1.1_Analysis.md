# Repository Analysis: C1.1 — Transaction Domain Model

## 1. Assessment
The current `src/domain/` directory contains `Transaction.gs`, `Category.gs`, `Money.gs`, and `TransactionType.gs`. These are rudimentary and need refactoring to support a robust domain-driven design that is independent of storage implementation.

## 2. Architecture Review
The domain model must remain agnostic of Google Sheets or other storage. 
- **Core Entities**: Transaction, Account, Category, Money (Value Object).
- **Relationships**: A Transaction relates to one Account and one Category.
- **Repository Interface**: `TransactionRepository` needs to be defined in `src/repository/` to support CRUD operations without implementing them yet.

## 3. Transaction Immutability & Repository Responsibility
- **Immutability Principle**: Financial transactions represent historical events. They should be treated as **immutable**. Physical deletion is prohibited as it destroys audit history and balances.
- **Lifecycle Management**: Future milestones should introduce business-oriented lifecycle states (e.g., `ACTIVE`, `REVERSED`, `CORRECTED`) rather than physical deletion.
- **Repository Responsibility**: The repository is solely responsible for persistence (save, findById, findAll). Business policies, including transaction lifecycle management (e.g., reversal), belong to the future domain/service layer. To enforce this, the `delete` operation is removed from the repository contract.

## 4. Risks
- Introducing coupling between the domain model and storage constraints.
- Over-engineering the domain model prematurely.

## 5. Acceptance Criteria
- [ ] Core domain entities (`Account`, `Transaction`, `Category`) are formally defined.
- [ ] Repository interfaces (`TransactionRepository`) are defined and storage-oriented (no business lifecycle methods).
- [ ] Domain logic uses `Money.gs` for financial calculations.
- [ ] Unit tests verify entity structures and repository interfaces.
- [ ] No implementation of persistence or business rules occurs.
